---
name: GlazeVault image cropper
description: In-app 4:5 crop preview contract and why saved crop matches every surface
---
- Custom `components/ImageCropper.tsx` replaced the OS cropper (`allowsEditing`/`aspect`). Pickers now use `allowsEditing:false, quality:1` and pass `{uri,width,height}` to the cropper; output flows unchanged through `persistPieceImage`.
- **Contract that guarantees "preview == saved, no re-crop":** every piece-image surface renders 4:5 — PotteryCard, add/edit form previews, and the detail hero (all `aspectRatio 4/5`, contentFit cover) — and fullscreen ImageViewer uses contentFit="contain". So exporting a precisely-4:5 cropped file matches all of them with zero further cropping. **Why:** if any surface used a different ratio with cover, it would silently re-crop and break the promise. If you change the card/hero ratio, change the cropper `aspectRatio` in lockstep.
- Crop math (cover baseline + pan/zoom → source pixels): `coverScale=max(fw/iw,fh/ih)`, `effScale=coverScale*s`, `cropW=fw/effScale`, `cropH=fh/effScale`, `originX=iw/2-(fw/2+tx)/effScale` (same for Y). Translation clamped to `±(baseW*s-fw)/2` so the frame is always covered.
- Cropping needs TRUE source pixel dims. Order: picker `asset.width/height` → `RNImage.getSize` → `measureViaManipulator` (decode) → notice + `onCancel()`. Never fabricate dims (fabricated dims are the only realistic preview/export mismatch path).
- **`RNImage.getSize` can HANG on some Android `content://` URIs (and chokes on some HEIC) — neither callback ever fires.** That deadlocked the cropper on a spinner with "Use Photo" disabled = the "stuck on last photo / won't let me add" alpha bug. Fix: race getSize against a ~2.5s timeout; on timeout/error fall back to `measureViaManipulator`; a `settled` flag + `clearTimeout` guard against double-resolution. **How to apply:** any dims-from-uri probe here must have a timeout escape, not rely on getSize calling back.
- **The cropper is rendered as TWO instances** (empty-state placeholder vs cover+strip), so adding the FIRST photo unmounts one instance and mounts the other. Async work in `handleConfirm` must guard setState with a `mountedRef`, and the dims effect must `setNatural(null)` on every open so a reused instance never frames the previous photo. Keep `processing` reset in a `finally` (mounted-guarded) so a failed crop re-enables retry instead of locking the buttons.
- expo-image-manipulator (~14.x, SDK54) new API: `ImageManipulator.manipulate(uri).crop(rect).resize({width}).renderAsync()` then `.saveAsync({compress,format})`. Works web (canvas) + native. Install via `expo install`, not the raw package tool, so the SDK-compatible version is picked.
- **WEB: saveAsync must request `base64:true` and the cropper hands back a `data:` URI**, not `result.uri`. The web manipulator output is a `blob:` URL, and `persistPieceImage` would `fetch(blob:)` it — which FAILS inside the sandboxed preview iframe → intermittent "Couldn't save photo". A `data:` URI is returned untouched by `persistPieceImage` (no fetch). On native keep `result.uri` (file path) so it's copied into `pieces/`. If web base64 is unexpectedly absent, throw (don't fall back to the blob uri).
